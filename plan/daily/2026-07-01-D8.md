## 5 条笔记

1. **为什么 W2 要换 Pydantic**：D5 的 `CallRecord` 是 `@dataclass`——字段齐了就行，**不校验**外部输入。模型返回的 JSON 是脏数据，必须 `model_validate` + catch `ValidationError`
2. **`BaseModel` 三板斧**：`model_validate(dict)` 进、`model_dump()` 出、`model_json_schema()` 给 API——类比 TS 的 `zod schema.parse()` + `.shape` 导出
3. **`Field(ge=0, le=1)`**：约束在**赋值时**检查，不是类型注解装饰；类比 C++ 没有内置 range，要靠构造函数 assert 或 `std::clamp` 后存
4. **`model_json_schema()` 的归宿**：D9 对比 JSON mode vs Structured Outputs；D10 ToolRegistry 的 `validate_args()` 同一套机制
5. **debug 脚本**：`scripts/d8_pydantic_debug.py`——先跑通 6 步再走 gateway 集成

### 核心 API 对照

| 方法 | 做什么 | TS 类比 |
|:---|:---|:---|
| `StructuredAnswer(...)` | 关键字构造，立即校验 | `schema.parse({...})` |
| `model_validate(dict)` | 从 LLM JSON 反序列化 | `JSON.parse` + zod parse |
| `model_dump()` | 对象 → dict | `{ ...obj }` |
| `model_dump_json()` | 对象 → JSON 字符串 | `JSON.stringify` |
| `model_json_schema()` | 导出 JSON Schema | 给 OpenAI `response_format` / `tools` |
| `ValidationError.errors()` | 逐字段错误定位 | zod `ZodError.issues` |

## 面试素材

> 结构化输出不是「让模型输出 JSON」这么简单——**解析 + 校验 + 可重试** 才是工程化。`model_json_schema()` 把约束声明一次，校验和 API schema 同源，避免手写两份定义漂移。
