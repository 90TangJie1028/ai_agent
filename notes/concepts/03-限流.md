# 限流与并发（D4 速查）

D3 是**被动扛**服务端 429（退避重试）；D4 是**主动控**自己发出的请求速率，避免打爆对方配额。

## 两个维度，别混


| 维度           | 控制什么              | 典型手段                   | Gateway 场景              |
| ------------ | ----------------- | ---------------------- | ----------------------- |
| **并发数**      | 同时 in-flight 的请求数 | `asyncio.Semaphore(N)` | 10 路同时 chat，但每路可能耗时 2s  |
| **QPS / 速率** | 单位时间放行多少请求        | 令牌桶 Token Bucket       | 每秒最多 5 次 create，第 6 次排队 |


并发 10 ≠ QPS 10：10 个请求若每个 100ms 完成，实际 QPS 可达 ~100。

## asyncio 最小集（Gateway 够用）

```python
import asyncio

async def fake_chat(i: int) -> str:
    await asyncio.sleep(0.1)  # 模拟网络 IO
    return f"ok-{i}"

async def main():
    tasks = [asyncio.create_task(fake_chat(i)) for i in range(10)]
    results = await asyncio.gather(*tasks)
    print(results)

asyncio.run(main())
```


| 概念            | 做什么            | 类比                                      |
| ------------- | -------------- | --------------------------------------- |
| `async def`   | 可挂起的协程函数       | TS `async function`；Cocos 里不是真并行，是协作式调度 |
| `await`       | 让出事件循环，等 IO 完成 | TS `await fetch()`                      |
| `create_task` | 并发启动多个协程       | `Promise.all` 前先 `Promise` 化            |
| `gather`      | 等一组任务全部结束      | `Promise.all(tasks)`                    |
| `asyncio.run` | 同步入口跑完整个事件循环   | Node 的顶层 `await` 需要包装层                  |


**差异**：Python 协程单线程事件循环；C++ 真多线程；TS 浏览器里也是单线程事件循环，语义最接近。

## Semaphore：并发上限

```python
sem = asyncio.Semaphore(3)  # 最多 3 个同时执行

async def bounded_call(i: int):
    # 进入这个代码块时会尝试“拿一个名额”；如果当前已满（比如只允许3个同时进行），就会自动 await 等前面有人退出，抢到空位才继续往下执行
    async with sem:          # 拿不到槽位就 await 排队
        return await fake_chat(i)
```

- `async with sem`：进入时 `acquire()`，退出时 `release()`；槽满则挂起等待。
- **类比 TS**：自己用 `p-limit(3)`；**C++**：`std::counting_semaphore`；**Cocos**：同时只允许 N 个资源加载回调 in-flight。

适用：**限制同时打出去的 HTTP 连接数**，防止内存/连接池爆掉。

## 令牌桶：QPS 上限

```
桶容量 = burst（突发上限，如 10）
每 Δt 补 rate 个令牌（如每秒补 5 个）
请求到来：有令牌 → 取 1 个放行；无令牌 → 等待或拒绝
```


| 参数      | 含义              |
| ------- | --------------- |
| `rate`  | 稳态 QPS（每秒补充令牌数） |
| `burst` | 桶容量，允许短时突发      |


与 D3 退避的关系：令牌桶是**预防** 429；退避是**事后**处理 429。两层都要有。

**类比**：

- TS：漏桶/令牌桶库（`bottleneck`）
- C++：网络库 rate limiter
- 游戏：技能 CD + 充能槽 — 稳态每秒回 1 点，槽满可连放 3 个

## D4 代码落点（计划）

```text
ratelimit.py     TokenBucket / RateLimiter；acquire() 阻塞或 async 等待
gateway.py       async_chat()；内部 Semaphore + 令牌桶
metrics.py       草案注释（D5 再实现 CallRecord）
tests/           mock 时间推进，断言 10 并发不超 QPS
```

### `async_chat` 伪代码

```python
async def async_chat(self, message: str, ...) -> ChatResult:
    async with self._concurrency_sem:       # 并发 ≤ 10
        await self._rate_limiter.acquire()    # QPS 限制
        return await asyncio.to_thread(      # 同步 SDK 丢线程池
            self.chat, message, ...
        )
```

`asyncio.to_thread`：把现有同步 `chat()` 包进线程，不阻塞事件循环 — 复用 D2/D3 代码，不必立刻改 OpenAI SDK 为 async 客户端。

## 测试思路（mock，不碰真 API）

1. **并发**：`Semaphore(10)` + 10 个 `create_task`，全部完成无异常。
2. **QPS**：mock `time.monotonic()` 或注入 clock；连续 acquire 20 次，统计 1 秒内放行数 ≤ rate。
3. **排队**：第 11 个并发请求开始时间应晚于前 10 个之一完成 — 证明在等槽位。

标记：默认单元测试 offline；真实 10 并发压测可标 `@pytest.mark.integration`。

## 与 D3 对比（面试用）


|      | D3 重试       | D4 限流                     |
| ---- | ----------- | ------------------------- |
| 谁触发  | 服务端 429/5xx | 客户端主动                     |
| 目的   | 故障恢复        | 流量整形                      |
| 手段   | 指数退避        | Semaphore + 令牌桶           |
| 失败形态 | 重试后仍失败抛异常   | 排队变长或 `RateLimitExceeded` |


## 5 条自检

1. 为什么并发控制和 QPS 控制要分开？→ 见上文「并发 10 ≠ QPS 10」
2. `async with Semaphore` 和 `threading.Semaphore` 区别？→ 前者不阻塞线程，挂起协程；后者阻塞 OS 线程
3. 令牌桶 `burst > rate` 有什么用？→ 允许短时突发，稳态仍受 rate 约束
4. 为什么用 `to_thread` 而不是重写 async SDK？→ 最小改动复用 D2/D3；后续可换 `AsyncOpenAI`
5. 限流满了该排队还是拒绝？→ Gateway 默认**排队**（用户体验）；可加 `max_wait` 超时后拒绝

## 面试一句话

> Gateway 出口两层护栏：Semaphore 卡并发、令牌桶卡 QPS，主动整形流量；同步 SDK 用 `asyncio.to_thread` 接入 async 接口，429 仍走 D3 退避。
