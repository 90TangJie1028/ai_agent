# 重试与超时（D3 速查）

大模型 API 是**不可靠外部服务**：限流、抖动、偶发 5xx 都常见。Gateway 要在 adapter 层统一处理，上层 `chat()` 无感。

## 超时

| 概念 | 说明 |
|------|------|
| 客户端超时 | 从发出请求到收到完整响应的最长等待时间 |
| 配置 | `GATEWAY_TIMEOUT_SEC`（默认 60），传给 OpenAI SDK 的 `timeout` |
| 行为 | SDK 抛 `APITimeoutError` → Gateway 转为 Python 标准 `TimeoutError` |
| 不重试 | 超时说明链路太慢或挂死，立即失败，避免堆积 |

```python
# openai_compat.py 核心
self._client.chat.completions.create(..., timeout=60.0)
```

## 可重试 vs 不可重试

| HTTP 码 | 含义 | 重试？ |
|---------|------|--------|
| 429 | 限流 / QPS 超限 | ✅ 指数退避 |
| 500/502/503/504 | 服务端临时故障 | ✅ |
| 连接失败 | 网络抖动 | ✅ |
| 401/403 | 鉴权失败 | ❌ Key 错了，重试无意义 |
| 400 | 请求参数错误 | ❌ 同样请求仍会失败 |

**幂等性**：Chat Completions 的「纯 user 消息、无工具副作用」可视为幂等——同一条 prompt 重试不会产生额外业务副作用（仅多耗 token）。带 tool call、写库等副作用操作后续要单独设计幂等键。

## 指数退避

```
第 0 次失败后等待 base × 2^0 = 0.5s
第 1 次失败后等待 base × 2^1 = 1.0s
第 2 次失败后等待 base × 2^2 = 2.0s
…封顶 max_delay（默认 8s）
```

默认策略（`RetryPolicy`）：

| 参数 | 默认值 |
|------|--------|
| `max_retries` | 3（共最多 4 次调用） |
| `base_delay_sec` | 0.5 |
| `max_delay_sec` | 8.0 |

## `@with_retry` 用法

```python
from model_gateway.retry import RetryPolicy, with_retry

@with_retry(RetryPolicy(max_retries=3))
def call_api():
    ...
```

装饰器用于**纯函数**；adapter 用显式循环以便绑定实例级 `RetryPolicy`。

## 与 D4 限流的区别

| | D3 重试 | D4 限流 |
|--|---------|---------|
| 触发方 | 服务端返回 429/5xx | 客户端主动控制 QPS |
| 目的 | 扛住对方限流/故障 | 避免自己打爆对方 |
| 手段 | 退避等待后重试 | 令牌桶 / Semaphore |

## QA
* 1. 429 为什么要退避？
    429表示被限流了，证明链路能通，可能请求频繁，这是API供应商服务器做出的策略，所以要退避
* 2. 401 为什么不重试？
    401/403 | 鉴权失败 | ❌ Key 错了，重试无意义 |

* 3. Chat 为什么说幂等可重试？工具调用后呢？

Chat 重试的安全前提：没有工具副作用，最多多耗 token；不是数学意义上的「结果一定相同」。

说白了 Chat 是一种是类似于代码中的 “赋值”操作，每次“赋值”都一样呗。所以从结果的角度来看，都是一样的，另外的，工具调用（也得看具体业务类似于查询历史数据（不可改变的）这种必然每次都一样，这种就可以幂等可重试），但是比如说的 类似 “累加/减”等 “累”的操作就不能随便重试呗

总结第1，2点问题，根据deepSeek文档（https://api-docs.deepseek.com/zh-cn/quick_start/error_codes）
4xx 基本都是客户端问题，根据对应错误码排查即可
5xx 基本API供应商服务器问题，500/503 也写的是「请等待后重试」，也可以切换供应商（通过之前的 gateWay）

## QA（自答 v2）

1. **429 为什么要退避？**
   429 = 速率到上限（TPM/RPM），链路通、服务活着。此时立刻重试会继续撞限流，
   可能加重封禁。退避是主动等待配额恢复，而不是跟服务端硬刚。

2. **401 为什么不重试？**
   401 = 鉴权失败，确定性错误，同请求重试不会变好。应快速失败，去查 Key/配置。
   4xx 大多是「修请求」问题，不是「多试几次」问题。（429 是例外）

3. **Chat 为什么说可重试？工具调用后呢？**
   纯 Chat（无工具）= 无副作用：最多多耗 token，一般不改变外部状态。
   注意：temperature>0 时回复未必相同，说的是「业务可接受重试」，不是严格幂等。
   工具调用看副作用：读操作（查历史）通常可重试；写操作（转账、累加、发邮件）不行。

4. **4xx / 5xx 怎么处理？（结合 DeepSeek 文档 + Gateway）**
   - 401/400/402/422：客户端问题 → 按错误码修，不重试
   - 429：限流 → 退避重试
   - 500/503：服务端临时故障 → 退避重试；仍失败再考虑 fallback 切供应商
