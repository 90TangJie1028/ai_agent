# ADR-001：Model Gateway 重试与超时策略

**状态**：已采纳  
**日期**：2026-06-24（W1 D3）  
**上下文**：P1 Model Gateway 对接 DeepSeek 等 OpenAI 兼容 API，需处理限流与瞬时故障。

## 决策

1. **超时**：默认 60s，经 `GATEWAY_TIMEOUT_SEC` 配置，透传至 OpenAI SDK `timeout` 参数。
2. **超时异常**：`APITimeoutError` 统一转为 `TimeoutError`，**不重试**。
3. **可重试状态码**：429、500、502、503、504，以及连接类错误。
4. **不可重试**：401、403、400 等客户端错误，立即向上抛出。
5. **退避算法**：指数退避，`delay = min(base × 2^attempt, max_delay)`，默认 base=0.5s、max=8s。
6. **最大重试次数**：3（即最多 4 次 API 调用）。
7. **实现位置**：重试逻辑在 `openai_compat._create_completion`；`retry.py` 提供可复用的 `@with_retry` 与判定函数。

## 理由

- **指数退避**：429 时立即连打会加重限流；逐步拉长间隔是行业惯例。
- **封顶延迟**：避免单次 chat 等待过长（8s × 多次仍可控）。
- **超时不重试**：超时多为网络或服务端无响应，盲目重试会拉长用户等待且难区分「慢」与「挂」。
- **幂等假设**：当前阶段仅 user 单轮消息，无副作用，可安全重试；W2 工具调用后需按工具类型再评估。

## 后果

### 正面

- 单元测试可 mock 429/超时，无需真实 API。
- 策略集中，D4 限流与 D5 metrics 可复用同一调用路径。

### 负面 / 风险

- 重试会增加 token 费用（若请求实际已成功但响应丢失，极端情况下可能重复计费——生产环境可考虑 request id）。
- 默认 60s 超时对长上下文生成可能偏紧，后续可按模型/场景分层配置。

## 备选方案（未采纳）

| 方案 | 放弃原因 |
|------|----------|
| 固定间隔重试（每次 1s） | 429 场景下仍易撞限流 |
| 超时也重试 | 用户感知延迟翻倍，难排查 |
| 在 Gateway 层重试 | Adapter 已封装 SDK，在最接近 IO 处重试信息最全 |
